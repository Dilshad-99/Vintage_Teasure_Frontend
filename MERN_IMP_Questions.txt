## JavaScript Fundamentals (Questions 1-30)

### Core Concepts

1. **What is the difference between `var`, `let`, and `const`?**

Question 1: What is the difference between var, let, and const?
Detailed Answer:
1. Scope

var: Function-scoped or globally-scoped. Ignores block scope (if, for, while blocks)
let and const: Block-scoped. Only accessible within the nearest enclosing block {}

// var - Function Scoped
function testVar() {
    if (true) {
        var x = 10;
    }
    console.log(x); // 10 - accessible outside block
}

// let - Block Scoped
function testLet() {
    if (true) {
        let y = 10;
    }
    console.log(y); // ReferenceError: y is not defined
}

2. Hoisting Behavior

var: Hoisted to the top and initialized with undefined
let and const: Hoisted but not initialized (Temporal Dead Zone)

// var hoisting
console.log(a); // undefined (declaration hoisted)
var a = 5;

// let/const hoisting - Temporal Dead Zone (TDZ)
console.log(b); // ReferenceError: Cannot access 'b' before initialization
let b = 10;

// TDZ explanation
{
    // TDZ starts
    // console.log(temp); // Error
    // console.log(temp); // Error
    let temp = 5; // TDZ ends
    console.log(temp); // 5
}

3. Re-declaration

var: Can be re-declared in the same scope
let and const: Cannot be re-declared

var x = 1;
var x = 2; // No error
console.log(x); // 2

let y = 1;
let y = 2; // SyntaxError: Identifier 'y' has already been declared

4. Re-assignment

var and let: Can be re-assigned
const: Cannot be re-assigned (but object properties can be modified)

var a = 1;
a = 2; // Allowed

let b = 1;
b = 2; // Allowed

const c = 1;
c = 2; // TypeError: Assignment to constant variable

// But with objects:
const obj = { name: 'John' };
obj.name = 'Jane'; // Allowed - modifying property
obj.age = 30; // Allowed - adding property
obj = {}; // TypeError - reassignment not allowed

5. Global Object Property

var: Creates property on global object (window/global)
let and const: Do not create global properties

var globalVar = 'I am global';
console.log(window.globalVar); // 'I am global' (in browser)

let globalLet = 'I am not on window';
console.log(window.globalLet); // undefined

When to Use:

const: Default choice - use for values that won't be reassigned
let: Use when you need to reassign values (loops, conditionals)
var: Avoid in modern code (legacy support only)

2. **Explain hoisting in JavaScript with examples.**

Question 2: Explain hoisting in JavaScript with examples.
Detailed Answer:
What is Hoisting?
Hoisting is JavaScript's behavior of moving declarations to the top of their scope during the compilation phase, before code execution. Only declarations are hoisted, not initializations.
Variable Hoisting
With var:

console.log(message); // undefined (not ReferenceError)
var message = 'Hello';
console.log(message); // 'Hello'

// How JavaScript interprets above code:
var message; // Declaration hoisted
console.log(message); // undefined
message = 'Hello'; // Initialization stays
console.log(message); // 'Hello'

With let and const:

console.log(temp); // ReferenceError: Cannot access 'temp' before initialization
let temp = 10;

// Explanation:
// let and const ARE hoisted, but they're in "Temporal Dead Zone" 
// until the line where they're initialized

Temporal Dead Zone (TDZ)
The time between entering scope and variable initialization where variable cannot be accessed.

{
    // TDZ starts for 'x'
    console.log(x); // ReferenceError
    console.log(x); // ReferenceError
    
    let x = 10; // TDZ ends, x is initialized
    
    console.log(x); // 10 - now accessible
}

Function Hoisting
Function Declarations - Fully Hoisted:

greet(); // "Hello!" - Works perfectly

function greet() {
    console.log('Hello!');
}

// Entire function is hoisted:
function greet() {
    console.log('Hello!');
}
greet(); // This is how JS sees it

Function Expressions - Not Hoisted:

sayHi(); // TypeError: sayHi is not a function

var sayHi = function() {
    console.log('Hi!');
};

// How JavaScript sees it:
var sayHi; // Declaration hoisted as undefined
sayHi(); // Calling undefined - TypeError
sayHi = function() { // Assignment happens here
    console.log('Hi!');
};

Arrow Functions - Not Hoisted:

greet(); // ReferenceError

const greet = () => {
    console.log('Hello');
};

Class Hoisting

const p = new Person(); // ReferenceError: Cannot access 'Person' before initialization

class Person {
    constructor(name) {
        this.name = name;
    }
}

// Classes are hoisted but in TDZ like let/const

Hoisting Order

Function declarations (highest priority)
Variable declarations
Function expressions (treated as variables)

console.log(typeof foo); // "function"

var foo = 'variable';

function foo() {
    return 'function';
}

console.log(typeof foo); // "string"

// How JS interprets:
function foo() { return 'function'; } // Function hoisted first
var foo; // Variable declaration (ignored - already exists)
console.log(typeof foo); // "function"
foo = 'variable'; // Assignment
console.log(typeof foo); // "string"

Practical Implications

// Bad practice - relies on hoisting
function calculateTotal() {
    price = 100; // Forgot 'var/let/const'
    var tax = 0.1;
    return price * (1 + tax);
}

calculateTotal(); // Works but creates global variable

// Good practice - declare at top
function calculateTotal() {
    var price = 100;
    var tax = 0.1;
    return price * (1 + tax);
}

3. **What are closures? Provide a practical use case.**

Question 3: What are closures? Provide a practical use case.
Detailed Answer:
What is a Closure?
A closure is a function that has access to:

Its own scope (variables defined between its curly braces)
Outer function's scope (even after outer function has returned)
Global scope

How Closures Work
When a function is created, it forms a closure around the variables in its scope. Even after the outer function finishes executing, the inner function retains access to those variables.
Basic Example

function outerFunction() {
    let outerVariable = 'I am from outer scope';
    
    function innerFunction() {
        console.log(outerVariable); // Accesses outer variable
    }
    
    return innerFunction;
}

const closureFunc = outerFunction();
closureFunc(); // "I am from outer scope"

// outerFunction has finished executing, but innerFunction
// still has access to outerVariable - this is closure

Why Closures Happen

function makeCounter() {
    let count = 0; // This variable is "closed over"
    
    return function() {
        count++; // Inner function accesses outer variable
        return count;
    };
}

const counter1 = makeCounter();
console.log(counter1()); // 1
console.log(counter1()); // 2
console.log(counter1()); // 3

const counter2 = makeCounter(); // New closure, new count
console.log(counter2()); // 1
console.log(counter2()); // 2

// Each counter has its own separate closure with its own 'count'

Practical Use Case 1: Data Privacy/Encapsulation

function createBankAccount(initialBalance) {
    let balance = initialBalance; // Private variable
    let transactionHistory = []; // Private variable
    
    return {
        deposit: function(amount) {
            if (amount > 0) {
                balance += amount;
                transactionHistory.push({ type: 'deposit', amount, balance });
                return balance;
            }
        },
        
        withdraw: function(amount) {
            if (amount > 0 && amount <= balance) {
                balance -= amount;
                transactionHistory.push({ type: 'withdraw', amount, balance });
                return balance;
            }
            return 'Insufficient funds';
        },
        
        getBalance: function() {
            return balance;
        },
        
        getHistory: function() {
            return [...transactionHistory]; // Return copy, not original
        }
    };
}

const myAccount = createBankAccount(1000);
console.log(myAccount.getBalance()); // 1000
myAccount.deposit(500); // 1500
myAccount.withdraw(200); // 1300

// Cannot access 'balance' directly
console.log(myAccount.balance); // undefined
// Balance is truly private through closure

function setupButtons() {
    for (let i = 1; i <= 3; i++) {
        const button = document.createElement('button');
        button.innerText = `Button ${i}`;
        
        // Closure captures the value of 'i'

        });
        
        document.body.appendChild(button);
    }
}

            console.log(`Button ${i} clicked`); // All show "Button 4"

function setupButtonsFixed() {

            const button = document.createElement('button');

            button.addEventListener('click', function() {
                console.log(`Button ${buttonNumber} clicked`);
            });
            
            document.body.appendChild(button);
        })(i); // IIFE creates new closure for each iteration
    }
}

Practical Use Case 4: Memoization/Caching

function createCache() {
    const cache = {}; // Private cache
    
    return {
        set: function(key, value) {
            cache[key] = value;
        },
        
        get: function(key) {
            return cache[key];
        },
        
        has: function(key) {
            return key in cache;
        },
        
        clear: function() {
            for (let key in cache) {
                delete cache[key];
            }
        }
    };
}

const myCache = createCache();
myCache.set('user1', { name: 'John', age: 30 });
console.log(myCache.get('user1')); // { name: 'John', age: 30 }

// Cache object is completely private through closure

Practical Use Case 5: Module Pattern

const Calculator = (function() {
    // Private variables and functions
    let history = [];
    
    function addToHistory(operation, result) {
        history.push({ operation, result, timestamp: new Date() });
    }
    
    // Public API
    return {
        add: function(a, b) {
            const result = a + b;
            addToHistory(`${a} + ${b}`, result);
            return result;
        },
        
        subtract: function(a, b) {
            const result = a - b;
            addToHistory(`${a} - ${b}`, result);
            return result;
        },
        
        getHistory: function() {
            return [...history];
        },
        
        clearHistory: function() {
            history = [];
        }
    };
})();

Calculator.add(5, 3); // 8
Calculator.subtract(10, 4); // 6
console.log(Calculator.getHistory());
// Cannot access 'history' or 'addToHistory' directly

4. **What is the difference between `==` and `===`?**

Question 4: What is the difference between == and ===?
Detailed Answer:
Core Difference

== (Loose/Abstract Equality): Compares values after type coercion
=== (Strict Equality): Compares values without type coercion (type AND value must match)

Type Coercion with ==
JavaScript converts operands to the same type before comparison.

// Number and String
console.log(5 == '5');    // true (string '5' converted to number 5)
console.log(5 === '5');   // false (different types)

// Boolean and Number
console.log(true == 1);   // true (true converted to 1)
console.log(true === 1);  // false (different types)
console.log(false == 0);  // true (false converted to 0)
console.log(false === 0); // false (different types)

// Empty String and Boolean
console.log('' == false);  // true (both converted to 0)
console.log('' === false); // false (different types)

// null and undefined
console.log(null == undefined);  // true (special case)
console.log(null === undefined); // false (different types)

// Object and Primitive
console.log([1] == 1);    // true ([1] converted to '1', then to 1)
console.log([1] === 1);   // false (different types)

Type Coercion Rules for ==

If types are same: Compare like ===
null and undefined: Equal to each other, nothing else
Number and String: Convert string to number
Boolean and anything: Convert boolean to number (true→1, false→0)
Object and primitive: Convert object to primitive using valueOf() or toString()

// Detailed examples:

// null and undefined
console.log(null == undefined);  // true
console.log(null == 0);          // false
console.log(undefined == 0);     // false

// String to Number conversion
console.log('123' == 123);       // true
console.log('0' == 0);           // true
console.log('' == 0);            // true

// Boolean conversion
console.log('1' == true);        // true ('1' → 1, true → 1)
console.log('0' == false);       // true ('0' → 0, false → 0)
console.log('true' == true);     // false ('true' → NaN, true → 1)

// Object to Primitive
console.log([1, 2] == '1,2');    // true ([1,2].toString() = '1,2')
console.log({} == '[object Object]'); // true

Surprising Cases with ==

// Tricky examples
console.log(' \t\n' == 0);       // true (whitespace → 0)
console.log([] == 0);            // true ([] → '' → 0)
console.log([] == false);        // true ([] → '' → 0, false → 0)
console.log([''] == '');         // true
console.log([0] == 0);           // true
console.log([1, 2] == '1,2');    // true

// Very confusing
console.log([] == ![]);          // true ([] → 0, ![] → false → 0)
console.log('' == [null]);       // true (both → '')

// NaN
console.log(NaN == NaN);         // false (NaN is not equal to anything)
console.log(NaN === NaN);        // false

Strict Equality === Examples

// No type coercion
console.log(5 === 5);            // true (same type, same value)
console.log(5 === '5');          // false (different types)
console.log(true === 1);         // false (different types)
console.log(null === undefined); // false (different types)
console.log(0 === false);        // false (different types)

// Objects (reference comparison)
const obj1 = { a: 1 };
const obj2 = { a: 1 };
const obj3 = obj1;

console.log(obj1 === obj2);      // false (different objects)
console.log(obj1 === obj3);      // true (same reference)

// Arrays
console.log([1, 2] === [1, 2]);  // false (different arrays)

// Special values
console.log(NaN === NaN);        // false (use Number.isNaN())
console.log(+0 === -0);          // true (but Object.is(+0, -0) is false)

When to Use Each
Use === (Strict Equality) - Default choice:

// Always use === for predictable behavior
if (userInput === '5') { }
if (count === 0) { }
if (isActive === true) { }

Use == (Loose Equality) - Very rare cases:

// Only when you specifically want to check null OR undefined
if (value == null) { 
    // Catches both null and undefined
    // Equivalent to: value === null || value === undefined
}

// This is the ONLY recommended use of ==

Best Practices

// ❌ Bad - Unexpected behavior
if (user.age == '18') { 
    // Matches both 18 and '18'
}

// ✅ Good - Clear intent
if (user.age === 18) {
    // Only matches number 18
}

// ❌ Bad - Confusing
if (isLoggedIn == 1) { }

// ✅ Good - Explicit
if (isLoggedIn === true) { }

// ❌ Bad - Multiple checks
if (value === null || value === undefined) { }

// ✅ Good - Only acceptable use of ==
if (value == null) { }

// ✅ Better - Modern approach
if (value ?? false) { } // Nullish coalescing

Object.is() - Even Stricter

// Object.is() is stricter than ===

console.log(Object.is(5, 5));           // true
console.log(Object.is(5, '5'));         // false

// Differences from ===:
console.log(NaN === NaN);               // false
console.log(Object.is(NaN, NaN));       // true

console.log(+0 === -0);                 // true
console.log(Object.is(+0, -0));         // false

Real-World Example

function validateUserInput(input, expected) {
    // ❌ Dangerous with ==
    if (input == expected) {
        return true;
    }
    
    // These would all match unexpectedly:
    // validateUserInput('0', false)   → true
    // validateUserInput('', 0)        → true
    // validateUserInput([], false)    → true
}

function validateUserInput(input, expected) {
    // ✅ Safe with ===
    if (input === expected) {
        return true;
    }
    // Only exact matches
}

5. **Explain the concept of "this" keyword in JavaScript.**

Question 5: Explain the concept of "this" keyword in JavaScript.
Detailed Answer:
What is this?
this is a special keyword in JavaScript that refers to the object that is executing the current function. Its value is determined by HOW a function is called, not where it's defined.
1. Global Context
In the global execution context, this refers to the global object.

// In Browser
console.log(this); // Window object
this.globalVar = 'I am global';
console.log(window.globalVar); // 'I am global'

// In Node.js
console.log(this); // global object (in REPL) or {} (in module)

// Strict mode
'use strict';
function showThis() {
    console.log(this); // undefined in strict mode
}
showThis();

2. Object Method Context
When a function is called as a method of an object, this refers to that object.

const person = {
    name: 'John',
    age: 30,
    greet: function() {
        console.log(this); // person object
        console.log(this.name); // 'John'
        console.log(this.age); // 30
    },
    
    getDetails: function() {
        return `${this.name} is ${this.age} years old`;
    }
};

person.greet(); 
// 'this' inside greet() refers to 'person'

// Nested objects
const user = {
    name: 'Alice',
    address: {
        city: 'New York',
        getCity: function() {
            console.log(this); // address object, not user
            return this.city;
        }
    }
};

user.address.getCity(); // 'New York'
// 'this' refers to the object immediately before the dot

const person = {
    name: 'John',
    age: 30,
    greet: function() {
        console.log(this); // person object
        console.log(this.name); // 'John'
        console.log(this.age); // 30
    },
    
    getDetails: function() {
        return `${this.name} is ${this.age} years old`;
    }
};

person.greet(); 
// 'this' inside greet() refers to 'person'

// Nested objects
const user = {
    name: 'Alice',
    address: {
        city: 'New York',
        getCity: function() {
            console.log(this); // address object, not user
            return this.city;
        }
    }
};

user.address.getCity(); // 'New York'
// 'this' refers to the object immediately before the dot

function showThis() {
    console.log(this); // Window (non-strict) or undefined (strict)
}

showThis();

// Common mistake:
const person = {
    name: 'John',
    hobbies: ['reading', 'coding'],
    showHobbies: function() {
        this.hobbies.forEach(function(hobby) {
            // 'this' here is undefined (strict) or Window (non-strict)
            console.log(this.name + ' likes ' + hobby); 
            // Error: Cannot read property 'name' of undefined
        });
    }
};

// Solution 1: Arrow function
const person1 = {
    name: 'John',
    hobbies: ['reading', 'coding'],
    showHobbies: function() {
        this.hobbies.forEach((hobby) => {
            console.log(this.name + ' likes ' + hobby); // Works!
        });
    }
};

// Solution 2: Save 'this' reference
const person2 = {
    name: 'John',
    hobbies: ['reading', 'coding'],
    showHobbies: function() {
        const self = this; // Save reference
        this.hobbies.forEach(function(hobby) {
            console.log(self.name + ' likes ' + hobby); // Works!
        });
    }
};

// Solution 3: Use bind()
const person3 = {
    name: 'John',
    hobbies: ['reading', 'coding'],
    showHobbies: function() {
        this.hobbies.forEach(function(hobby) {
            console.log(this.name + ' likes ' + hobby);
        }.bind(this)); // Bind 'this'
    }
};

4. Arrow Functions
Arrow functions DO NOT have their own this. They inherit this from the enclosing lexical scope.

const obj = {
    name: 'Arrow Object',
    regularFunc: function() {
        console.log(this.name); // 'Arrow Object'
    },
    arrowFunc: () => {
        console.log(this.name); // undefined (inherits global 'this')
    }
};

obj.regularFunc(); // 'Arrow Object'
obj.arrowFunc(); // undefined

// Practical use of arrow functions:
const timer = {
    seconds: 0,
    start: function() {
        // Arrow function inherits 'this' from start()
        setInterval(() => {
            this.seconds++;
            console.log(this.seconds);
        }, 1000);
    }
};

timer.start(); // 1, 2, 3, 4... (works correctly)

// With regular function (wrong):
const timer2 = {
    seconds: 0,
    start: function() {
        setInterval(function() {
            this.seconds++; // 'this' is Window, not timer2
            console.log(this.seconds); // NaN
        }, 1000);
    }
};

5. Constructor Functions
When a function is called with new, this refers to the newly created object.

function Person(name, age) {
    // 'this' refers to new empty object {}
    this.name = name;
    this.age = age;
    this.greet = function() {
        console.log(`Hi, I'm ${this.name}`);
    };
    // Implicitly returns 'this'
}

const john = new Person('John', 30);
console.log(john.name); // 'John'
john.greet(); // 'Hi, I'm John'

// What 'new' does:
// 1. Creates empty object: const this = {}
// 2. Sets prototype: this.__proto__ = Person.prototype
// 3. Executes function with 'this' bound to new object
// 4. Returns 'this' (unless function explicitly returns object)

// Without 'new':
const jane = Person('Jane', 25); // Wrong!
console.log(jane); // undefined
console.log(window.name); // 'Jane' (polluted global scope!)

6. Explicit Binding: call(), apply(), bind()
call() - Invoke function with specific this and arguments separately:

function introduce(greeting, punctuation) {
    return `${greeting}, I'm ${this.name}${punctuation}`;
}

const person1 = { name: 'Alice' };
const person2 = { name: 'Bob' };

console.log(introduce.call(person1, 'Hello', '!')); 
// "Hello, I'm Alice!"

console.log(introduce.call(person2, 'Hi', '.')); 
// "Hi, I'm Bob."

// Practical use: Borrowing methods
const obj1 = {
    name: 'Object 1',
    printName: function() {
        console.log(this.name);
    }
};

const obj2 = { name: 'Object 2' };

obj1.printName.call(obj2); // 'Object 2'

apply() - Same as call() but arguments as array:

function introduce(greeting, punctuation) {
    return `${greeting}, I'm ${this.name}${punctuation}`;
}

const person = { name: 'Charlie' };

console.log(introduce.apply(person, ['Hey', '?'])); 
// "Hey, I'm Charlie?"

// Practical use: Finding max in array
const numbers = [5, 6, 2, 3, 7];
const max = Math.max.apply(null, numbers);
console.log(max); // 7

// Modern alternative with spread:
const max2 = Math.max(...numbers);

bind() - Creates new function with bound this:

function introduce(greeting) {
    return `${greeting}, I'm ${this.name}`;
}

const person = { name: 'David' };

const boundIntroduce = introduce.bind(person);
console.log(boundIntroduce('Hello')); // "Hello, I'm David"
console.log(boundIntroduce('Hi')); // "Hi, I'm David"

// Practical use: Event handlers
const button = {
    content: 'Click me',
    click: function() {
        console.log(this.content + ' was clicked');
    }
};

// Without bind:
// document.getElementById('btn').addEventListener('click', button.click);
// Error: 'this' would be the button element, not our object

// With bind:
document.getElementById('btn').addEventListener('click', button.click.bind(button));
// Now 'this' correctly refers to button object

// Partial application with bind:
function multiply(a, b) {
    return a * b;
}

const double = multiply.bind(null, 2); // First argument fixed to 2
console.log(double(5)); // 10
console.log(double(8)); // 16

const triple = multiply.bind(null, 3);
console.log(triple(5)); // 15

7. Class Methods
In ES6 classes, this refers to the instance.

class Person {
    constructor(name) {
        this.name = name;
    }
    
    greet() {
        console.log(`Hi, I'm ${this.name}`);
    }

    greetArrow = () => {
        console.log(`Hi, I'm ${this.name}`);
    }
}

9. Priority/Precedence of this Binding
From highest to lowest priority:

new binding - Constructor call
Explicit binding - call(), apply(), bind()
Implicit binding - Method call
Default binding - Global object or undefined

function test() {
    console.log(this.name);
}

const obj1 = { name: 'Object 1' };
const obj2 = { name: 'Object 2' };

// Default binding
test(); // undefined (strict mode)

// Implicit binding
obj1.test = test;
obj1.test(); // 'Object 1'

// Explicit binding
test.call(obj2); // 'Object 2'

// Explicit binding wins over implicit
const boundTest = test.bind(obj2);
obj1.test = boundTest;
obj1.test(); // 'Object 2' (not 'Object 1')

// new binding wins over explicit
function Person(name) {
    this.name = name;
}

const boundPerson = Person.bind(obj1);
const person = new boundPerson('New Person');
console.log(person.name); // 'New Person' (not 'Object 1')

6. **What are arrow functions and how do they differ from regular functions?**

Question 6: What are arrow functions and how do they differ from regular functions?
Detailed Answer:
What are Arrow Functions?
Arrow functions are a shorter syntax for writing functions introduced in ES6. They use the => syntax.
Syntax Variations

// Regular function
function add(a, b) {
    return a + b;
}

// Arrow function - full syntax
const add = (a, b) => {
    return a + b;
};

// Arrow function - implicit return (single expression)
const add = (a, b) => a + b;

// Single parameter - parentheses optional
const double = x => x * 2;

// No parameters - parentheses required
const sayHello = () => console.log('Hello');

// Returning object literal - wrap in parentheses
const makePerson = (name, age) => ({ name: name, age: age });

// Or with shorthand
const makePerson2 = (name, age) => ({ name, age });

// Multiple statements - use braces and explicit return
const complexFunc = (x, y) => {
    const sum = x + y;
    const product = x * y;
    return sum + product;
};

Key Difference 1: Lexical this Binding
Arrow functions don't have their own this. They inherit it from the enclosing scope.

// Regular function - own 'this'
const regularObj = {
    name: 'Regular',
    greet: function() {
        console.log(this.name); // 'Regular'
    }
};

// Arrow function - no own 'this'
const arrowObj = {
    name: 'Arrow',
    greet: () => {
        console.log(this.name); // undefined (global 'this')
    }
};

regularObj.greet(); // 'Regular'
arrowObj.greet(); // undefined

// Practical example - Timer
function Timer() {
    this.seconds = 0;
    
    // Regular function - wrong
    setInterval(function() {
        this.seconds++; // 'this' is window/global
        console.log(this.seconds); // NaN
    }, 1000);
}

function Timer2() {
    this.seconds = 0;
    
    // Arrow function - correct
    setInterval(() => {
        this.seconds++; // 'this' is Timer2 instance
        console.log(this.seconds); // 1, 2, 3...
    }, 1000);
}

function Timer3() {
    this.seconds = 0;
    const self = this; // Old solution
    
    setInterval(function() {
        self.seconds++;
        console.log(self.seconds);
    }, 1000);
}

// Real-world React example
class MyComponent extends React.Component {
    constructor() {
        super();
        this.state = { count: 0 };
    }
    
    // Regular method - needs binding
    handleClickRegular() {
        this.setState({ count: this.state.count + 1 });
    }
    
    // Arrow function - auto-bound
    handleClickArrow = () => {
        this.setState({ count: this.state.count + 1 });
    }
    
    render() {
        return (
            <div>
                {/* Need to bind regular method */}
                <button onClick={this.handleClickRegular.bind(this)}>
                    Regular
                </button>
                
                {/* Arrow function works directly */}
                <button onClick={this.handleClickArrow}>
                    Arrow
                </button>
            </div>
        );
    }
}

Key Difference 2: No arguments Object

    console.log(arguments[0]); // First argument
    console.log(arguments.length); // Number of arguments
}

regularFunc(1, 2, 3); // Arguments [1, 2, 3]

// Arrow function - no 'arguments'

    console.log(arguments); // ReferenceError
};

// Solution: Use rest parameters
const arrowWithRest = (...args) => {
    console.log(args); // Array [1, 2, 3]
    console.log(args[0]); // 1
    console.log(args.length); // 3
};

arrowWithRest(1, 2, 3);

// Arrow function in nested scope inherits outer 'arguments'
function outer() {
    const inner = () => {
        console.log(arguments); // outer's arguments
    };
    inner();
}

outer(1, 2, 3); // [1, 2, 3]

Key Difference 3: Cannot be Used as Constructor

// Regular function - can be constructor
function Person(name) {
    this.name = name;
}

const person1 = new Person('John');
console.log(person1.name); // 'John'

// Regular function - explicit return
function add(a, b) {
    return a + b; // Must use 'return'
}

// Arrow function - implicit return (single expression)
const addArrow = (a, b) => a + b; // No 'return' needed

undefined: Absence of assignment - variable exists but has no value

8. **Explain event bubbling and event capturing.**

Question 8: Explain event bubbling and event capturing.Detailed Answer:What is Event Propagation?
When an event occurs on a DOM element, it doesn't just trigger on that element - it propagates through the DOM tree in three phases:
Capturing Phase (top to bottom) - from window to target
Target Phase - event reaches the target element
Bubbling Phase (bottom to top) - from target back to window

Window
  ↓ Capturing
Document
  ↓
<html>
  ↓
<body>
  ↓
<div>
  ↓
<button> ← Target (Click happens here)
  ↑
<div>
  ↑ Bubbling
<body>
  ↑
<html>
  ↑
Document
  ↑
Window

Event Bubbling (Default Behavior)Events "bubble up" from the target element to its ancestors.

<!-- HTML Structure -->
<div id="grandparent">
    <div id="parent">
        <button id="child">Click Me</button>
    </div>
</div>

// Event Bubbling Example
document.getElementById('child').addEventListener('click', function() {
    console.log('Child clicked');
});

document.getElementById('parent').addEventListener('click', function() {
    console.log('Parent clicked');
});

document.getElementById('grandparent').addEventListener('click', function() {
    console.log('Grandparent clicked');
});

// When you click the button:
// Output:
// "Child clicked"
// "Parent clicked"
// "Grandparent clicked"
// (From innermost to outermost)

Event Capturing (Capture Phase)Events "trickle down" from the window to the target element. Must explicitly enable using third parameter true.

// Event Capturing Example
document.getElementById('child').addEventListener('click', function() {
    console.log('Child clicked');
}, true); // true = capture phase

document.getElementById('parent').addEventListener('click', function() {
    console.log('Parent clicked');
}, true); // true = capture phase

document.getElementById('grandparent').addEventListener('click', function() {
    console.log('Grandparent clicked');
}, true); // true = capture phase

// When you click the button:
// Output:
// "Grandparent clicked"
// "Parent clicked"
// "Child clicked"
// (From outermost to innermost)

Complete Event Flow (Capturing + Bubbling)

// Mixed: Some capturing, some bubbling
document.getElementById('grandparent').addEventListener('click', function() {
    console.log('Grandparent (Capturing)');
}, true);

document.getElementById('parent').addEventListener('click', function() {
    console.log('Parent (Bubbling)');
}, false); // false = bubble phase (default)

document.getElementById('child').addEventListener('click', function() {
    console.log('Child (Target)');
});

document.getElementById('parent').addEventListener('click', function() {
    console.log('Parent (Capturing)');
}, true);

// When you click the button:
// Output:
// "Grandparent (Capturing)"  ← Capturing phase
// "Parent (Capturing)"       ← Capturing phase
// "Child (Target)"           ← Target phase
// "Parent (Bubbling)"        ← Bubbling phase

Stopping Event Propagation
1. event.stopPropagation() - Stops further propagation

document.getElementById('child').addEventListener('click', function(event) {
    console.log('Child clicked');
    event.stopPropagation(); // Stop bubbling
});

document.getElementById('parent').addEventListener('click', function() {
    console.log('Parent clicked'); // This won't execute
});

document.getElementById('grandparent').addEventListener('click', function() {
    console.log('Grandparent clicked'); // This won't execute
});

9. **What is the difference between synchronous and asynchronous JavaScript?**

Question 9: What is the difference between synchronous and asynchronous JavaScript?
Detailed Answer:
Synchronous vs Asynchronous - Core Concept
Synchronous: Code executes line by line, each operation must complete before the next one starts (blocking).
Asynchronous: Code can start an operation and move to the next line before the operation completes (non-blocking).
Synchronous Execution

console.log('Start');

function syncTask() {
    console.log('Sync task 1');
    console.log('Sync task 2');
    console.log('Sync task 3');
}

syncTask();

console.log('End');

// Output (guaranteed order):
// Start
// Sync task 1
// Sync task 2
// Sync task 3
// End

Blocking Example:

console.log('Start');

// Simulated long operation (bad practice)
function longRunningTask() {
    const end = Date.now() + 3000; // 3 seconds from now
    while (Date.now() < end) {
        // Blocking - browser freezes
    }
    console.log('Long task complete');
}

longRunningTask(); // Blocks for 3 seconds

console.log('End'); // Has to wait

// During those 3 seconds:
// - Browser is frozen
// - User can't interact
// - UI doesn't update

Asynchronous Execution

console.log('Start');

setTimeout(() => {
    console.log('Async task');
}, 2000);

console.log('End');

// Output:
// Start
// End
// Async task (after 2 seconds)

// Explanation:
// 1. "Start" logged
// 2. setTimeout scheduled (doesn't wait)
// 3. "End" logged immediately
// 4. After 2 seconds, "Async task" logged

How JavaScript Handles Async Code
JavaScript is single-threaded but handles async operations using:

Call Stack - Where code executes
Web APIs - Browser features (setTimeout, fetch, DOM events)
Callback Queue - Queue for callbacks
Event Loop - Moves callbacks from queue to stack

console.log('1');

    console.log('2');
}, 0); // Even with 0ms delay!

console.log('3');

// Output: 1, 3, 2

// Explanation:
// 1. console.log('1') - executes immediately
// 2. setTimeout - goes to Web API, callback queued
// 3. console.log('3') - executes immediately
// 4. Call stack empty, event loop moves callback to stack
// 5. console.log('2') - executes

// ❌ Slow - Sequential (1s + 1s + 1s = 3s)

    const ids = [1, 2, 3];
    const users = [];
    
    for (const id of ids) {

        users.push(user);

    const ids = [1, 2, 3];
    const promises = ids.map(id => fetchUser(id));
    const users = await Promise.all(promises);

}

10. **What are callbacks and what is callback hell?**

Question 10: What are callbacks and what is callback hell?Detailed Answer:What is a Callback Function?A callback is a function passed as an argument to another function, which is then invoked (called back) inside the outer function to complete some kind of routine or action. Callbacks are fundamental to asynchronous programming in JavaScript.Theoretical Foundation:

Callbacks implement the Observer Pattern - they allow functions to notify other functions when an operation completes
They enable Inversion of Control - you give control to another function to execute your code at the appropriate time
They are first-class citizens in JavaScript - functions can be passed as arguments, returned from functions, and assigned to variables
Types of Callbacks1. Synchronous Callbacks
Executed immediately within the calling function's execution context.javascript// Synchronous callback - executes immediately
function processArray(arr, callback) {
    for (let i = 0; i < arr.length; i++) {
        callback(arr[i], i); // Callback executes synchronously
    }
}

processArray([1, 2, 3], (item, index) => {
    console.log(`Item ${index}: ${item}`);
});

// Array methods use synchronous callbacks
const doubled = [1, 2, 3].map(num => num * 2);
// The callback executes immediately for each elementTheoretical Characteristics:

Executes in the same call stack frame
Blocks further execution until complete
No context switching between execution
Predictable execution order
2. Asynchronous Callbacks
Executed later, after an asynchronous operation completes.javascript// Asynchronous callback - executes later
function fetchData(url, callback) {
    // Simulating network request
    setTimeout(() => {
        const data = { id: 1, name: 'User' };
        callback(null, data); // Callback executes asynchronously
    }, 1000);
}

fetchData('/api/user', (error, data) => {
    if (error) {
        console.error('Error:', error);
    } else {
        console.log('Data:', data);
    }

Deferred execution - callback placed in event queue
Non-blocking - allows other code to execute
Executed when call stack is empty (via Event Loop)
Unpredictable execution timing (depends on async operation)
Callback Pattern - Error-First ConventionNode.js popularized the "error-first" callback pattern:javascript// Error-first callback signature
function asyncOperation(params, callback) {
    // callback signature: (error, result)
    // First parameter: error (null if no error)
    // Second parameter: result (undefined if error occurred)
}

// Implementation
function readFile(filename, callback) {
    fs.readFile(filename, 'utf8', (err, data) => {
        if (err) {
            callback(err, null); // Error case
            return;
        }
        callback(null, data); // Success case
    });
}

// Usage

        return;
    }
    console.log('File contents:', data);

Consistency: All async functions follow same pattern
Error Handling: Forces developers to handle errors explicitly
Type Safety: Clear distinction between error and success states
What is Callback Hell?Callback hell (also called "Pyramid of Doom") occurs when callbacks are nested within callbacks multiple levels deep, making code:

Difficult to read
Hard to maintain
Error-prone
Difficult to debug
The Problem - Theoretical Perspective:When you have multiple sequential asynchronous operations that depend on each other, traditional callbacks lead to deeply nested code structures:javascript// Classic Callback Hell Example

11. **Explain Promises and their states.**

Question 11: Explain Promises and their states.
Detailed Answer:
What is a Promise? - Theoretical Foundation
A Promise is an object representing the eventual completion or failure of an asynchronous operation. It's a proxy for a value that is not yet known when the Promise is created.
Formal Definition:
A Promise is a monad - a design pattern from functional programming that:

Wraps a value (or future value)
Provides methods to transform the wrapped value (.then())
Enables composition of operations
Handles side effects in a controlled manner

Mathematical Analogy:
Think of a Promise like a "future" or "deferred" value:

Present: const x = 5 (value available now)
Future: const x = Promise.resolve(5) (value available later)

The Three States of a Promise
A Promise is a finite state machine with exactly three mutually exclusive states:
1. Pending (Initial State)

The operation has not completed yet
Promise is neither fulfilled nor rejected
This is the transient state - all Promises start here
Can transition to either Fulfilled or Rejected

2. Fulfilled (Success State)

The operation completed successfully
Promise has a result value
This is a terminal state - cannot change to any other state
Also called "Resolved"

3. Rejected (Failure State)

The operation failed
Promise has a reason (usually an Error object)
This is a terminal state - cannot change to any other state
Represents an error or exception

javascript// State transition diagram (theoretical):
//
//       Pending
//      /       \
//     /         \
//  Fulfilled  Rejected
//  (terminal) (terminal)
//
// Once settled (fulfilled or rejected), state is immutable
Promise State Transitions - Detailed
javascript// Creating a Promise - starts in Pending state
const promise = new Promise((resolve, reject) => {
    // State: Pending
    
    setTimeout(() => {
        const success = Math.random() > 0.5;
        
        if (success) {
            resolve('Success value'); // Transition to Fulfilled
            // State is now: Fulfilled
            // Result: 'Success value'
        } else {
            reject(new Error('Failure reason')); // Transition to Rejected
            // State is now: Rejected
            // Reason: Error('Failure reason')
        }
    }, 1000);
});

// Accessing promise state (internal - not directly accessible)
// Promise {
//   [[PromiseState]]: "pending" | "fulfilled" | "rejected"
//   [[PromiseResult]]: undefined | value | reason
// }

Once a Promise settles (becomes fulfilled or rejected), its state and value are immutable:
javascriptconst promise = new Promise((resolve, reject) => {

    resolve('Second value'); // Ignored - already fulfilled
    reject(new Error('Error')); // Ignored - already fulfilled
});

    console.log(value); // "First value" - only the first settlement counts
});

// It cannot change once settled - provides reliability and predictability

Internal Mechanics - Generator-Based Implementation
Theoretical Foundation: Async/await is implemented using generator functions internally.
javascript// What you write:

    const a = await promise1();
    const b = await promise2(a);
    return b;

// Conceptual transformation (simplified):
function example() {
    return spawn(function* () {
        const a = yield promise1();
        const b = yield promise2(a);
        return b;
    });
}

function spawn(generatorFunc) {
    return new Promise((resolve, reject) => {
        const generator = generatorFunc();
        
        function step(nextValue) {
            const result = generator.next(nextValue);
            
            if (result.done) {
                resolve(result.value);
                return;
            }
            
            Promise.resolve(result.value)
                .then(value => step(value))
                .catch(error => reject(error));
        }
        
        step();
    });
}
Theoretical Insights:

Generator State: Maintains function state between yield points
Iterative Unwrapping: Each Promise is unwrapped before continuing
Automatic Chaining: Engine handles Promise chaining automatically

javascriptasync function retry(fn, maxAttempts = 3, delay = 1000) {
    for (let attempt = 1; attempt <= maxAttempts; attempt++) {

            return await fn();
        } catch (error) {
            if (attempt === maxAttempts) {
                throw error;
            }
            
            console.log(`Attempt ${attempt} failed, retrying...`);
            await new Promise(resolve => setTimeout(resolve, delay));
        }
    }
}

javascriptasync function mapWithConcurrencyLimit(items, fn, limit) {
    const results = [];
    const executing = [];
    
    for (const item of items) {
        const promise = fn(item).then(result => {
            executing.splice(executing.indexOf(promise), 1);
            return result;
        });
        
        results.push(promise);
        executing.push(promise);
        
        if (executing.length >= limit) {
            await Promise.race(executing);
        }
    }
    
    return Promise.all(results);
}

```
## **Question 13: What is the Event Loop in JavaScript?**

**Detailed Answer:**

### **What is the Event Loop? - Theoretical Foundation**

The Event Loop is the **concurrency model** that allows JavaScript to perform non-blocking I/O operations despite being single-threaded. It's the mechanism that coordinates the execution of code, collection of events, and processing of queued tasks.

**Fundamental Concepts:**
- **Single-Threaded**: JavaScript has only one call stack
- **Non-Blocking**: Asynchronous operations don't block execution
- **Event-Driven**: Responds to events via callback execution
- **Cooperative Multitasking**: Code voluntarily yields control

**Theoretical Model:**
JavaScript runtime consists of several components that work together:
```
┌───────────────────────────┐
│   JavaScript Runtime      │
├───────────────────────────┤
│  ┌─────────────────────┐  │
│  │    Call Stack       │  │  (Execution context)
│  └─────────────────────┘  │
│  ┌─────────────────────┐  │
│  │    Heap             │  │  (Memory allocation)
│  └─────────────────────┘  │
└───────────────────────────┘
            ↕
┌───────────────────────────┐
│      Web APIs             │
│  (Browser/Node.js)        │
│  - setTimeout             │
│  - fetch                  │
│  - DOM events             │
│  - I/O operations         │
└───────────────────────────┘
            ↓
┌───────────────────────────┐
│   Callback Queues         │
├───────────────────────────┤
│  Microtask Queue          │  (Higher priority)
│  - Promise callbacks      │
│  - queueMicrotask         │
│  - MutationObserver       │
├───────────────────────────┤
│  Macrotask Queue          │  (Lower priority)
│  - setTimeout             │
│  - setInterval            │
│  - I/O callbacks          │
│  - UI rendering           │
└───────────────────────────┘
            ↑
       Event Loop
      (Coordinates)
```

### **The Event Loop Algorithm - Formal Specification**

**Pseudocode:**
```
while (eventLoop.waitForTask()) {
    // 1. Execute one macrotask (if available)
    const task = macrotaskQueue.dequeue();
    execute(task);
    
    // 2. Execute ALL microtasks
    while (microtaskQueue.hasTask()) {
        const microtask = microtaskQueue.dequeue();
        execute(microtask);
    }
    
    // 3. Render (if browser and needed)
    if (isRepaintTime()) {
        render();
    }
}

Check Call Stack: If not empty, execute current code
Check Microtask Queue: Process ALL microtasks
Check Macrotask Queue: Process ONE macrotask
Repeat from step 2
